# CS-255-DriverPass-Portfolio
DriverPass system analysis and design project including business requirements and system design documentation.
The DriverPass project was about designing a cloud-based system for a driver training company called DriverPass. The client wanted a system that would help students better prepare for their DMV driving test by combining online practice exams with in-person driving lessons. The system needed to allow customers to register, purchase training packages, schedule and cancel lessons, and access online practice materials. It also needed to support multiple user roles, including students, instructors, office staff, administrators, and IT personnel. The goal was to improve DMV pass rates while also making scheduling and management more efficient for the company. One thing I feel I did particularly well was organizing the business requirements and translating the client’s needs into clear system requirements. In the Business Requirements Document, I clearly outlined the system purpose, objectives, functional requirements, and nonfunctional requirements. Then in the System Design Document, I built on that by defining technical requirements such as cloud hosting, database structure, role-based access control, and security features like HTTPS and account lockout.

If I could revise one part of my work, I would improve the UML diagrams. While they covered the required components, I would refine them to show clearer relationships between users and system processes. I would also make the sequence diagram more detailed to better show the flow of data between the user interface, application layer, and database. Improving visual clarity would make the design easier for stakeholders to understand. When interpreting user needs, I focused on what problems DriverPass was trying to solve. The company wanted to reduce DMV test failures and manage scheduling more efficiently. I translated those needs into system features such as online scheduling, instructor and vehicle assignment, progress tracking, and activity logs. It is important to consider user needs when designing because the system exists to solve their problem. If the design does not reflect the user’s workflow and goals, then even a technically strong system will fail in practice. When I approach designing software, I start by understanding the problem before thinking about technology. I break the system down into components such as users, data, processes, and security. I also think about scalability and future growth. In the future, I would use techniques like requirement traceability matrices, user story mapping, and more detailed UML modeling to ensure every requirement is connected directly to a design element. I would also involve user feedback earlier in the design process to validate assumptions before finalizing the architecture. Overall, this project strengthened my understanding of how business requirements connect directly to system design and how important it is to think from both a technical and user perspective.à
